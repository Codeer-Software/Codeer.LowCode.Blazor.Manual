# 在庫管理テンプレート

中小規模の小売・卸売・製造業が、商品の入出庫・棚卸し・現在庫を一元管理し、適正在庫を維持するための雛形アプリ。Excel・紙台帳から脱却して、複数人同時編集・自動集計・発注点アラート付きのデジタル管理へ移行するためのたたき台です。

内部名: `InventoryManagementTemplate`

## 画面構成

| メニュー | 用途 |
|---|---|
| ホーム | KPI 8 枚（要発注 / 遅延発注 / 発注中 / 発注残金額 / 在庫切れ / 今月入出庫 / 総在庫金額）+ 月次入出庫推移グラフ |
| 在庫一覧 | 商品 × 倉庫の現在庫数を一覧表示。検索・絞り込みの起点 |
| 要発注品 | 現在庫が発注点を下回った商品の一覧。「一括発注」ボタンで仕入先 × 倉庫別の発注書を自動生成 |
| 発注 | 発注書の作成・確定・追跡。仮登録 → 発注中 → 一部入庫 → 完了 のステータス管理。発注書から入庫を起こすボタン付き |
| 入庫 / 出庫 / 棚卸し | 各伝票の CRUD。ヘッダ + 明細リストの統一レイアウト |
| マスタ（商品 / 仕入先 / 倉庫） | 各マスタの管理 |

### ホーム（ダッシュボード）

KPI カードは 2 段 4 列の計 8 枚。左上にアクション系（要発注・遅延）、右列に金額系（円）を並べた配置です。

**Row 1（アクション系・進行系）**

| ウィジェット | 内容 | 条件 |
|---|---|---|
| 要発注件数（赤） | 発注点を下回っている商品 × 倉庫の組合せ数 | `current_stock < reorder_point` |
| 遅延発注件数（赤） | 希望納期超過の未完了発注書 | `status IN ('発注中','一部入庫') AND desired_delivery_date < 今日` |
| 発注中件数（青） | 進行中の発注書件数 | `status IN ('発注中','一部入庫')` |
| 発注残金額（橙） | 未完了発注の残金額合計 | `SUM((quantity − received_quantity) × unit_price)` |

**Row 2（在庫状況・実績系）**

| ウィジェット | 内容 | 条件 |
|---|---|---|
| 在庫切れ件数（赤） | 在庫が 0 以下の商品 × 倉庫の組合せ数 | `current_stock ≤ 0` |
| 今月の入庫件数（青） | 当月の入庫伝票件数 | `receiving_date` が当年月 |
| 今月の出庫件数（紫） | 当月の出庫伝票件数 | `shipping_date` が当年月 |
| 総在庫金額（緑） | 全在庫の評価額合計 | `SUM(current_stock × unit_price)` |

KPI 下に **月次入出庫推移グラフ**（直近 6 ヶ月の入庫量・出庫量の棒グラフ）。さらにその下に要発注品リストを内包しています。

「要発注件数」「遅延発注件数」は KPI 右に `→` ボタンを表示し、対応する一覧（要発注品ページ / 発注一覧の遅延フィルタ）へ即遷移できます。

### 要発注品リスト

<img src="../../Image/web/templates/template_inventory_reorder.png" alt="要発注品リスト（発注点を下回った商品 + 一括発注ボタン）" style="border: 1px solid #ccc;" width="900">

現在庫が発注点を下回った商品が「不足」「推奨」付きで並びます。右下の「一括発注」ボタンを押すと、仕入先 × 倉庫別の **仮登録** 発注書が一括作成されます。推奨数量は `(発注点 + 安全在庫) − 現在庫` の式で自動算出。

## データモデル

マスタ 3 / トランザクションヘッダ 4 / トランザクション明細 4 / 在庫スナップショット 1 の **計 12 テーブル**。各テーブルは `id`（PK） / `created_at` / `updated_at` を共通で持ちます（画面非表示）。

### テーブル一覧

| 区分 | テーブル | 用途 |
|---|---|---|
| マスタ | `product` | 商品マスタ |
| マスタ | `supplier` | 仕入先マスタ |
| マスタ | `warehouse` | 倉庫マスタ |
| トランザクション | `purchase_order` | 発注ヘッダ |
| トランザクション | `purchase_order_detail` | 発注明細 |
| トランザクション | `receiving` | 入庫ヘッダ |
| トランザクション | `receiving_detail` | 入庫明細 |
| トランザクション | `shipping` | 出庫ヘッダ |
| トランザクション | `shipping_detail` | 出庫明細 |
| トランザクション | `stocktaking` | 棚卸しヘッダ |
| トランザクション | `stocktaking_detail` | 棚卸し明細 |
| スナップショット | `inventory` | 商品 × 倉庫の現在庫 |

### マスタ系

| テーブル | 主な列 |
|---|---|
| `product`（商品） | code（UNIQUE）/ name / category / unit / unit_price / reorder_point（発注点）/ safety_stock（安全在庫）/ supplier_id |
| `supplier`（仕入先） | name / contact_name / phone / email / address |
| `warehouse`（倉庫） | name / address / note |

### トランザクション系（ヘッダ + 明細）

| テーブル | 主な列 |
|---|---|
| `purchase_order`（発注ヘッダ） | order_no（'PO-YYYYMMDD-NNN'、確定時に採番）/ order_date / desired_delivery_date / supplier_id / warehouse_id / status / version（楽観ロック）|
| `purchase_order_detail`（発注明細） | product_id（内部参照）/ product_code, product_name（伝票時点保持）/ quantity / unit_price / **received_quantity**（入庫済数量。分納で加算）|
| `receiving`（入庫ヘッダ） | receiving_date / warehouse_id / supplier_id / note |
| `receiving_detail`（入庫明細） | product_id / product_code, product_name（伝票時点保持）/ quantity / unit_price |
| `shipping`（出庫ヘッダ） | shipping_date / warehouse_id / shipping_to（出庫先テキスト）/ note |
| `shipping_detail`（出庫明細） | product_id / product_code, product_name（伝票時点保持）/ quantity |
| `stocktaking`（棚卸しヘッダ） | stocktaking_date / warehouse_id / note |
| `stocktaking_detail`（棚卸し明細） | product_id / inventory_id（引当）/ book_stock（帳簿在庫、自動引当）/ actual_stock（実在庫、手入力）/ difference（自動計算）|

### 在庫スナップショット

| テーブル | 主な列 |
|---|---|
| `inventory`（在庫） | product_id / warehouse_id / current_stock（現在庫数） / version（楽観ロック）|

商品 × 倉庫 で 1 行ずつ。入出庫・棚卸し登録のたびに **同じトランザクションで自動更新**。

### 関係図（簡略）

```
product ─┬──→ purchase_order_detail ←── purchase_order ──→ supplier
         ├──→ receiving_detail     ←── receiving      ──→ supplier
         ├──→ shipping_detail      ←── shipping
         ├──→ stocktaking_detail   ←── stocktaking
         └──→ inventory ──→ warehouse
```

すべての明細から `product_id` で商品を参照。伝票（ヘッダ）から `warehouse_id` で倉庫を参照。明細は `product_code` / `product_name` も持ち、**伝票時点の値を保持**します（マスタが後で変わっても過去伝票は崩れない）。

## 業務フロー

### 導入時（最初の一度だけ）

1. **マスタ整備**: 倉庫・仕入先・商品を登録（商品には発注点と安全在庫を必ず設定）
2. **初期在庫の登録**: 既存在庫を「棚卸し」で各倉庫の実在庫として一括登録（または初回の入庫伝票として登録）

### 日次

- 商品が入荷したら **入庫を登録**（発注書から「入庫登録」ボタンで起こすと商品・数量・倉庫が自動セット）
- 商品が出ていくときに **出庫を登録**（顧客出荷、社内払い出し、廃棄、倉庫間移動の出荷など）
- 発注書の進捗は **発注一覧** で確認

### 週次〜月次

- **要発注品** ページで発注点を下回った商品をチェック → 「一括発注」で仕入先 × 倉庫別の仮登録発注書を一括作成 → 発注確定して仕入先へ
- ホームの **KPI と月次推移グラフ** で在庫金額・入出庫・発注残金額・遅延発注などの動きを把握

### 四半期〜年次

- **棚卸し** を実施して帳簿在庫と実在庫のズレを補正（倉庫単位で実施可能、営業時間外推奨）

## 発注書のステータス遷移

```
（新規作成）
    ↓
[ 仮登録 ]  ──「発注確定」──→ [ 発注中 ]  ──「キャンセル」──→ [ キャンセル ]
    │                            │
   「破棄」                       │ 入庫登録（一部）
    ↓                            ↓
  （削除）                    [ 一部入庫 ]
                                 │
                                 │ 入庫登録（全数完了）
                                 ↓
                              [ 完了 ]
```

| ステータス | 編集可否 | 表示されるボタン |
|---|---|---|
| 仮登録 | 全項目編集可 | 登録 / 発注確定 / 破棄 |
| 発注中 | 備考のみ | 登録 / キャンセル / 入庫登録 |
| 一部入庫 | 備考のみ | 登録 / 入庫登録 |
| 完了 | 備考のみ | 登録 |
| キャンセル | 備考のみ | 登録 |

「発注確定」押下時に **発注番号 `PO-YYYYMMDD-NNN` を自動採番**（当日最大連番 +1、3 桁ゼロ埋め）してステータスを「発注中」に変更。仮登録中は `order_no` 列は NULL。

### 発注書から入庫を起こす

「発注中」「一部入庫」状態の発注書を開いて **入庫登録** ボタンを押すと、入庫モジュールがダイアログで開きます。

- 入庫日: 当日
- 倉庫: 発注書の納品先倉庫
- 仕入先: 発注書の仕入先
- 備考: `発注 PO-YYYYMMDD-NNN から入庫起こし`
- 明細: 各商品の **残数量（発注数量 − 入庫済数量）** を数量にセット、発注単価を仕入単価にコピー

検品結果に応じて数量を修正して「登録」を押すと、入庫登録 + 発注ステータスの自動更新が **1 つのトランザクションで実行** されます。全明細の入庫済数量が発注数量に達した時点で **「完了」** へ自動遷移、未達なら **「一部入庫」**。

## 在庫が更新される仕組み

入庫・出庫・棚卸しの登録ボタンを押すと、伝票（ヘッダ + 明細）の保存と在庫テーブルの更新が **1 つの処理としてまとめて実行** されます。

- **成功**: 伝票がすべて保存され、各明細の数量分だけ在庫が更新される
- **途中で失敗**: 伝票も在庫も全てキャンセル → 登録前の状態に戻る

「伝票は残ったのに在庫は古いまま」「在庫は動いたのに伝票がない」という不整合は構造上発生しません。

### 入出庫は相対更新（楽観ロックなし）

入庫・出庫は「現在の在庫数に対して数量を加減する」**相対更新**で処理。複数人が同じ商品・倉庫に対して同時に操作しても、処理の順序によらず最終的な在庫数は一致します（例: 100 個に対して +50 と −30 を同時実行すると、どちらが先でも結果は 120 個）。このため楽観的ロックは不要。

### 棚卸しは絶対更新（楽観ロック付き）

棚卸しは「現在の在庫数を実在庫数で **上書き** する」**絶対更新**。入出庫の相対更新と違って、他ユーザーの更新を後から打ち消すリスクがあるため、**楽観的ロック（バージョンチェック）** で保護しています。

棚卸し画面で商品コードを入力した瞬間、在庫テーブルから現在バージョンが取得されて棚卸し明細に保持されます。登録時に保持バージョンと現在バージョンが一致するかチェック:

- 一致 → 在庫上書き + バージョン +1（成功）
- 不一致 → 「更新に失敗しました」エラー（伝票も在庫も保存されない）

**推奨運用**: 棚卸しは営業時間外実施。中小企業では十分対応できます。

### 伝票は登録後に修正・削除できない

入庫・出庫・棚卸し伝票は、**一度登録したら修正も削除もできない** 設計です（事実の記録 / 監査連動 / 不正防止のため）。

入力ミスは **逆伝票** で打ち消します。

| 元の伝票 | 訂正方法 |
|---|---|
| 入庫を間違えた | 同じ商品・倉庫・数量で **出庫** を登録（打ち消し）→ 正しい内容で再度入庫 |
| 出庫を間違えた | 同じ商品・倉庫・数量で **入庫** を登録（打ち消し）→ 正しい内容で再度出庫 |
| 棚卸しを間違えた | 正しい実在庫で **新しい棚卸し** を登録（最新の棚卸しが優先） |

訂正の履歴も伝票として残るため、「いつ・誰が・何を訂正したか」が追跡可能。マスタ（商品 / 仕入先 / 倉庫）は通常通り編集・削除できます。発注書は仮登録中のみ全項目編集可、それ以降は備考のみ。

## 作るときのポイント

CLB で同等のテンプレートを 1 から組むときに重要な設計判断:

### 在庫を専用テーブルで保持する

「在庫数を `SUM(入庫) − SUM(出庫)` で集計クエリ」にすると伝票が増えるほど遅くなり、発注点判定もダッシュボードも待ち時間が伸びます。**`inventory` テーブルに 1 行ずつ持ち、伝票登録時にトリガー的に更新** する設計にすれば、画面表示・発注点判定・在庫金額計算がすべて高速。

### 明細のスナップショット保持

`purchase_order_detail` / `receiving_detail` / `shipping_detail` には `product_id`（FK）と一緒に `product_code` / `product_name` を持たせます。商品コード / 名前が後で変わっても **過去伝票の表示が崩れない**。FK は集計用、スナップショット文字列は表示用、と役割を分けるのがポイント。

### 商品コード入力で自動引当（連動入力）

明細で商品コードを直接入力したとき、`SelectField` の Candidates 動的取得や `ModuleSearcher` で商品マスタを引いて、商品名・単価・帳簿在庫を自動セットします。詳細は [連動入力パターン](../patterns/input_patterns.md)。

### 入庫起こしダイアログを「同じ受入モジュールの別 DetailLayout」で作る

専用の「入庫ダイアログ」モジュールを別に作るのではなく、`receiving` モジュールに **ダイアログ用 DetailLayout** を追加して `ShowDialog` で呼びます。テーブル / フィールドを共有できるので、ダイアログから「登録」しても通常の入庫伝票と同じ仕組みで在庫に反映されます。

### 1 トランザクションでの伝票 + 在庫更新

入庫明細の `OnSubmitAsync` などで `inventory` の `current_stock` を更新する SQL を発行するとき、**伝票本体の Submit と同じ DB トランザクション**で実行する必要があります。CLB の `ExecuteSqlField` / `OnAfterSubmitAsync` はデフォルトで同一トランザクションに乗ります。

### 楽観ロックは絶対更新側にだけ付ける

`inventory.version` は棚卸し（上書き）の安全装置として使い、入出庫（加減算）側ではバージョンチェックをスキップします。両方に付けると相対更新側で頻繁にコンフリクトしてしまうため。詳細は [楽観ロックパターン](../patterns/optimistic_lock.md)。

### 発注番号は確定時に採番（仮登録中は NULL）

`order_no` 列を NOT NULL UNIQUE にしてしまうと「仮登録に番号を付ける？」問題が出ます。**NULL 許容にしておき、確定スクリプトで `PO-YYYYMMDD-NNN` を採番** する設計にすると、仮登録の破棄時にも番号を消費せず済みます。

## カスタマイズのポイント

- 項目追加はデザイナ GUI から（コーディング不要）
- 商品カテゴリは TextField で運用（マスタ化したい場合はモジュール追加で対応）
- 単位（個 / 箱 / kg 等）も TextField なので自由に入力
- 発注書のステータス遷移ロジックを自社運用に合わせて追加・削除する場合はモジュールスクリプト（`.mod.cs`）を編集
- 「破棄」「キャンセル」確認ダイアログのテキストは商談スクリプト内の文字列

## 想定業種

物理的な商品在庫を扱う業種であれば適用可能。

| 業種 | 使用ケース |
|---|---|
| 小売業 | 店舗・倉庫間の在庫管理、発注タイミングの把握 |
| 卸売業 | 多品種の入出庫管理、得意先別出荷管理 |
| 製造業（資材・部品） | 原材料・部品の入出庫、安全在庫の維持 |
| EC 事業者 | 商品在庫の一元管理、複数倉庫間の在庫把握 |
| 建設・工事業 | 資材・工具の在庫管理、現場ごとの払い出し記録 |
| 飲食チェーン本部 | 食材・備品の在庫管理、店舗別払い出し |

規模感としては **商品点数 100〜2,000 点、倉庫 1〜5 拠点、月間伝票 200〜2,000 件** に最適。
