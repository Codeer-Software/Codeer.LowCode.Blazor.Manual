# 営業支援 (SFA) テンプレート

中小規模の営業組織が、商談の進捗と活動履歴を共有しつつ、パイプライン全体の健康度をひと目で把握できるようにするための雛形アプリ。Excel・SaaS から脱却する SFA（Sales Force Automation）のたたき台です。

内部名: `SFATemplate`

## 画面構成

| メニュー | 用途 |
|---|---|
| ホーム | 対象年月セレクタ + 要対応サマリー + KPI カード + フェーズ別パイプライン + 担当者別受注/活動グラフ |
| 要対応商談 | 基準日時点で「遅延（受注予定日超過）」または「停滞（30日以上活動なし）」の商談を抽出表示 |
| 商談 | 商談の CRUD。詳細画面に活動履歴を内包表示 + 活動履歴追加ダイアログ |
| マスタ（業種 / 顧客 / 営業担当者） | 業種マスタ、顧客（会社）+ 顧客担当者、自社の営業メンバー |

### ホーム（ダッシュボード）の構成

最上部に **対象年月セレクタ**（DateField の年月モード）を配置。値を変えると配下の集計が一斉に切り替わります。

| ウィジェット | 種類 | 内容 |
|---|---|---|
| 要対応サマリー | カード | 対象年月の月末時点で遅延（受注予定日超過）または停滞（30日以上活動なし）の商談件数 |
| KPI カード（件数） | 数値 + Donut | 受注確定 vs 予定の件数を数値とドーナツチャートで対比 |
| KPI カード（金額） | 数値 + Donut | 受注確定 vs 予定の金額を数値とドーナツチャートで対比 |
| フェーズ別商談本数 | 棒グラフ | 対象月に受注予定だった商談のフェーズ別本数（受注含む・失注除く・全フェーズ表示） |
| フェーズ別金額 | 棒グラフ | 対象月に受注予定だった商談のフェーズ別予測金額合計 |
| 営業担当者別受注金額 | 棒グラフ | 対象月の受注金額（`status='受注'` の `expected_amount` 合計）を担当者別に降順表示 |
| 担当者別活動件数 | グループ棒グラフ | 対象月の活動履歴を担当者 × 種別（訪問/電話/メール/Web会議）で集計（活動 0 件の担当者も表示） |

進行中商談・直近活動の単独リストはダッシュボードには載せない設計（「集計は Home、明細は各画面」の役割分担）。商談詳細の活動履歴がそれを兼ねます。

### 商談詳細画面のレイアウト

営業が活動履歴を確認しながら作業しやすいよう、商談本体の項目を開閉式で整理しています。

- **常時表示（最上段）**: 案件名 / 顧客 / ステータス / 登録ボタン
- **顧客追加ボタン**: 顧客フィールド横のアイコンボタン。クリックで顧客登録ダイアログを開き、未登録の顧客をその場で登録 → そのまま商談の顧客に設定できる
- **基本情報セクション**（開閉式 `IsExpandable`）: 商談責任者 / 受注予定日 / 確度 / 予測金額 / 受注失注理由 / 商談メモ
- **活動履歴**: `DetailListField` で「行」レイアウトを使ったカード風一覧（活動日時 Desc）
- **活動履歴追加ボタン**: クリックで活動履歴モジュールをダイアログ表示。OK で活動履歴 ListField に新規行追加（保存は商談本体の登録ボタンで一括）

## データモデル

商談を中心に、活動履歴で営業活動を時系列に記録する **6 テーブル構成**。各テーブルは `id`（PK） / `created_at` / `updated_at` を共通で持ちます。

### テーブル一覧

| テーブル | 用途 |
|---|---|
| `deal` | 商談 |
| `activity` | 活動履歴（商談に紐付く訪問・電話・メール等の記録） |
| `customer` | 顧客（会社） |
| `contact` | 顧客担当者（顧客に紐付く個人） |
| `industry` | 業種マスタ |
| `salesperson` | 営業担当者（自社メンバー） |

### 各テーブルの主な列

| テーブル | 主な列 |
|---|---|
| `deal`（商談） | name（案件名）/ status（フェーズ）/ probability（確度）/ expected_amount（予測金額）/ expected_close_date（受注予定日）/ won_lost_reason / memo / customer_id（→ customer）/ owner_id（→ salesperson） |
| `activity`（活動履歴） | deal_id（→ deal）/ type（訪問/電話/メール/Web会議）/ activity_at / performer_id（→ salesperson、担当者）/ contact_id（→ contact、やり取り相手）/ memo |
| `customer`（顧客） | name（会社名）/ industry_id（→ industry）/ address / phone / web |
| `contact`（顧客担当者） | customer_id（→ customer）/ name / department / title / email / phone |
| `industry`（業種） | name |
| `salesperson`（営業担当者） | name / department / email |

### 関係図

```
industry ←── customer ──┬── contact ←──┐
                        │              │
                        └── deal ──── activity
                                       │
                        salesperson ←──┤  (performer_id)
                                       │
                        salesperson ←──┘  (deal.owner_id)
```

`deal.owner_id` で商談責任者、`activity.performer_id` で活動の担当者、`activity.contact_id` で顧客側のやり取り相手を表します。

### 活動履歴の役割と絞り込み

活動履歴は商談ごとに紐付き、各活動には **「担当した営業」と「やり取りした顧客担当者」** を記録します。やり取り相手の選択候補は **商談に紐付く顧客の担当者だけ** に絞り込まれます（`activity.deal_id.customer_id.Value` をドット参照してフィルタを構成）。

1 活動 = 1 相手の方針（複数人とのやり取りは活動を分けるか、メモ欄に併記）。

## 業務フロー

### 導入時

1. **マスタ準備**: 営業担当者・業種を登録（業種は運用中も追加可能）
2. **顧客登録**: 会社情報（業種選択あり）を登録 → 詳細画面で顧客担当者を追加

### 日々の運用

- **商談登録**: 顧客と商談責任者を選んで新規作成 → 保存後、詳細画面で活動履歴を追加
- **活動記録**: 商談詳細画面の「活動履歴追加」ボタンからダイアログを開き、訪問・電話・メール等のログを記入。OK で商談の活動履歴リストに追加され、商談の登録ボタンで一括保存
- **要対応商談の確認**: メニュー「要対応商談」で、基準日時点で遅延・停滞の商談を一覧化。デフォルトの基準日は翌月末、変更すれば任意時点の状態を確認可能
- **状況把握**: ホームのダッシュボードで対象年月を選択 → 要対応件数・KPI・パイプライン・営業担当者別受注を月単位で確認

### 商談新規作成時のフロー

商談を新規作成する際は、まず「顧客（会社）」と「商談責任者」を選んで保存。保存後に詳細画面が開き、活動履歴を追加していきます。各活動の「やり取り相手」候補は、商談に紐付く顧客の担当者に絞り込まれます。

## 作るときのポイント

CLB で同等のテンプレートを 1 から組むときに重要な設計判断:

### 「集計は Home、明細は各画面」の分業

ダッシュボードに進行中商談・直近活動の **単独リストを置かない** ことで、ホームを「対象月の状況サマリー」に集中させています。明細を見たいときは商談一覧・商談詳細・要対応商談ページで確認する流れ。同じデータを 2 つの画面で持つと「どちらが正？」になるので、役割を明確に分けます。

### 対象年月セレクタによる全チャート連動

`Home.mod.cs` の `OnAfterInitialization` で対象年月を初期化、対象年月フィールドの `OnAfterValueChangedAsync` で全チャート（QueryField）の `Reload()` を一括呼び出し。これにより 1 つのセレクタ変更で 6〜7 ウィジェットが一斉に再描画されます。

ロード制御として、初期化中は `AllowLoad = false` にして、初期化が終わってから `true` に切替 → 一斉に最初のロードが走る、というパターンを使うと「対象年月セット中に部分的に古い月のデータが見える」事象を防げます。

### 活動履歴の「やり取り相手」を顧客に応じて絞り込む

`activity` の `contact_id`（LinkField → contact）の検索フィルタに `activity.deal_id.customer_id.Value == contact.customer_id` のような条件を書きます。**ドット参照で親モジュールの値を引いて絞り込む** のが連動入力パターンの典型形。詳細は [連動入力パターン](../patterns/input_patterns.md)。

### 商談詳細に活動履歴を内包 + 追加ダイアログで AddRow

活動履歴は商談詳細の `DetailListField` に直接並べ、追加は商談スクリプトの `OnClick` で「活動履歴の DetailLayout（ダイアログ用）」を `ShowDialog` で開く → OK 時に `ListField.AddRow()` で行追加 → 保存は商談本体の登録ボタンで一括、という流れ。ダイアログ内では仮データを編集するだけで DB へは書き込まないので、キャンセル時に何も残らない。

### 要対応商談の「基準日」初期化

要対応商談リストの基準日を **翌月末** に初期化することで、「来月末時点で遅延・停滞しているもの」を予防的に把握できます。`OnAfterInitialization` で翌月末を計算してセット、`OnAfterValueChangedAsync` で QueryField を Reload。

### KPI カードの「数値 + Donut」二重表示

件数 / 金額の KPI を **数値とドーナツチャートで対比** することで、達成率が直感的に伝わります。Donut は「受注確定」「残り（予定）」の 2 値を渡すだけ。CLB の ApexCharts バインディング (`Codeer.LowCode.Bindings.ApexCharts`) でシンプルに組めます。

### フェーズ定義は JSON 編集で自社向けに

商談ステータス（フェーズ）は `deal.mod.json` の SelectField Candidates に列挙されています。自社の商談プロセス（例: リード → アポ → ヒアリング → 提案 → 見積 → クロージング → 受注）に合わせて編集できます。フェーズ別パイプラインの並び順もここで決まります。

## カスタマイズのポイント

- 項目追加はデザイナ GUI から（コーディング不要）
- フェーズの定義・色は商談モジュールの JSON。自社の商談プロセスに合わせて編集可能
- ApexCharts の種類（棒／円／折れ線）も差し替え容易
- 1 活動 = 1 相手の方針を超えたい場合は `activity_contact`（中間テーブル）を追加

## 想定業種

法人営業（BtoB）を行う組織であれば業種を問わず適用可能。

| 業種 | 使用ケース |
|---|---|
| BtoB SaaS / IT サービス | リード〜契約までの商談進行、フェーズ別パイプライン管理 |
| 製造業（営業部門） | 法人顧客への提案・見積〜受注の進捗管理 |
| 商社・卸 | 多取引先・多商品案件の並行進行管理 |
| コンサルティング | 提案フェーズの長い案件の状況可視化 |
| 人材紹介・派遣 | 法人クライアントごとの案件・進捗管理 |
| 不動産（法人向け） | 物件提案〜契約までの商談・連絡履歴の集約 |
| 広告・代理店 | クライアント別の提案案件、競合状況管理 |

規模感としては **営業 5〜30 名程度、同時 30〜100 商談程度** に最適。
