# 1.2.57

スクリプト言語の仕様整理にともない、実行時挙動が変化する修正が入りました。
既存スクリプトに影響する可能性がある項目の一覧です。

---

## 1. `*=` が正しく掛け算になりました（バグ修正）

`*=` が足し算として動作していたバグを修正しました。

```csharp
int x = 3;
x *= 5;
// 変更前: x は 8（3 + 5 として計算されていた）
// 変更後: x は 15（3 * 5）
```

ローカル変数、フィールド、配列要素すべての `*=` 代入が対象です。
旧挙動に依存して `+=` のつもりで `*=` と書いていた場合は、`+=` に書き換えてください。

---

## 2. 後置 `++` / `--` の式としての戻り値を C# 準拠に修正（バグ修正）

式として使ったときの戻り値が変わりました。

```csharp
int x = 5;
int y = x++;
// 変更前: y は null（x は 6 にはなる）
// 変更後: y は 5（C# 準拠で旧値を返す、x は 6）

int z = ++x;
// 変更前: z は null
// 変更後: z は 6（C# 準拠で新値を返す）
```

文として単独で使っている行（`x++;` のみ）の挙動は変わりません。
旧挙動 (`y` が null) に依存しているコードは以下のように書き換えてください。

```csharp
int y = x;
x++;
```

---

## 3. オーバーロード解決で継承距離が近いほうが選ばれるようになりました（C# 準拠）

多段継承の場合に、より派生した型のオーバーロードが選ばれるようになりました。

```csharp
// Puppy : Dog : AnimalBase の継承関係
// f(object) / f(AnimalBase) / f(Dog) が定義されていて Puppy を渡したとき
// 変更前: f(AnimalBase) が選ばれていた
// 変更後: f(Dog) が選ばれる（最も派生した型）
```

同名の .NET メソッドオーバーロードで多段継承がある場合に選択結果が変わる可能性があります。

---

## 4. 明示型ローカル変数への再代入で型変換されるようになりました

`int x; double y;` など明示的に型を指定したローカル変数への再代入時に、
宣言した型に合わせて自動的に変換されるようになりました。

```csharp
int x = 0;
x = "5";     // 変更前: x は "5"（string のまま）  / 変更後: x は 5（int に変換）
x = "abc";   // 変更前: x は "abc"（string のまま）/ 変更後: x は null（変換失敗で null）
x = 1.7;     // 変更前: x は 1.7（double のまま）  / 変更後: x は 2（int、四捨五入）

double y = 1;
y = 2;       // 変更前: y は 2（int に変化）      / 変更後: y は 2.0（double 維持）
```

- `var` 宣言は従来どおり型を固定しないため、この変更の影響を受けません。
- 明示型変数に変換不能な値を代入していた箇所は、結果が `null` になります。
- 意図せず型が変わっていた箇所を救済する修正ですが、変換失敗で `null` になるケースに注意してください。

---

## 5. モジュール変数への再代入でも型変換されるようになりました

4 と同じルールがモジュールトップレベルで宣言した `int x;` などの変数にも適用されます。
他モジュールからの代入（`mod.x = value;`）も含まれます。

```csharp
// モジュール A のトップレベルで int counter; と宣言
// 他モジュールから A.counter = "5"; とした場合
// 変更前: counter は "5"（string のまま）
// 変更後: counter は 5（int に変換）
```

---

## 6. スクリプト関数のオーバーロード解決が最適マッチになりました

スクリプト内で同名関数を複数定義した際、
**先頭から順に適用可能なものが選ばれる**方式から、
**引数の型に最も一致するものが選ばれる**方式に変更されました。

```csharp
int F(int a) { return 1; }
int F(string a) { return 2; }
return F("5");
// 変更前: 1 が返る（F(int) が先頭で適用可能だったため）
// 変更後: 2 が返る（F(string) が完全一致で勝つ）
```

.NET メソッドのオーバーロードと同じ規則で解決されるようになりました。
宣言順を前提としていた箇所は、引数の型を明確にしてください。

---

## 7. スクリプト関数の完全同型重複は実行時エラーになりました

同名・同仮引数型のスクリプト関数定義が複数ある状態でその関数を呼び出すと、
**「関数の重複定義があります」** という実行時エラーになります。

```csharp
int F(int a) { return 1; }
int F(int a) { return 2; }
// 変更前: 最初の F が静かに使われていた
// 変更後: F を呼び出すと実行時エラー
```

意図せず同シグネチャの関数を重複定義していた場合は動かなくなります。
デバッグ時の混乱を避けるための変更です。一方を削除するか、引数の型を変えてください。

---

## 8. 名前解決順が変わり、Field / Layout / モジュール変数がサービス / 型名より優先されるようになりました

識別子の解決順が変更されました。

**変更前:**

1. ローカル変数
2. サービス / 型
3. モジュールに宣言されたメンバー（Field / Layout / モジュール変数 / CurrentUser）

**変更後:**

1. ローカル変数
2. モジュールに宣言されたメンバー（Field / Layout / モジュール変数 / CurrentUser）
3. サービス / 型

```csharp
// モジュールに Field "DateTime" がある場合
return DateTime.Today;
// 変更前: 型の DateTime.Today が返る（Field は隠れていた）
// 変更後: Field "DateTime" が返るため、.Today にアクセスするとエラー
```

Field / Layout / モジュール変数の名前が、
`Math`, `DateTime`, `Logger`, `Guid`, `TimeSpan` など
サービス名や型名と同名の場合のみ挙動が変わります。

該当する名前を使っている場合は、Field 等の名前を変更してください（例: `DateTime` → `CurrentDate`）。

## 影響度まとめ

| # | 変更 | 影響度 |
|---|---|---|
| 1 | `*=` バグ修正 | 中（誤った挙動に依存していた場合のみ） |
| 2 | 後置 `++` / `--` の戻り値 | 低（式として使う箇所は少ない） |
| 3 | オーバーロード継承距離 | 低（多段継承 + 同名オーバーロードが必要） |
| 4 | 明示型ローカル変数の再代入で型変換 | 中〜高（型違い代入を多用していると影響） |
| 5 | モジュール変数の再代入で型変換 | 中〜高 |
| 6 | スクリプト関数の最適マッチ解決 | 中（同名オーバーロードがある場合） |
| 7 | 完全同型重複で実行時エラー | 中（重複を意図せず書いていたコードは要修正） |
| 8 | 名前解決順の変更（Field 等 > サービス / 型） | 低（Field 名と予約語的な名前の衝突が必要） |

---

## 移行ガイド

| 壊れるケース | 修正方法 |
|---|---|
| `x *= n` を足し算として使っていた | `x += n` に書き換える |
| `y = x++` の `y` が null 前提 | `int y = x; x++;` に分離 |
| `int x = 0; x = "abc";` が `"abc"` になる前提 | `var x` にするか、意図通りの変換コードを書く |
| Field 名に `Math` / `DateTime` 等を使用 | Field 名を変更（例: `CurrentDate`） |
| 完全同型の同名関数定義 | 一方を削除、またはシグネチャを変更 |
| 宣言順前提の同名関数 | 引数の型を明確にして意図したオーバーロードが選ばれるようにする |
